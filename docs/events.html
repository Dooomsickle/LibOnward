Events

Events are an incredibly powerful feature of C# that allow us to run code when certain things happen. We can both make our own and subscribe to IL2CPP events.

Defining Our Own

We can create an event simply by using the event keyword. You must define it as a delegate.

What are delegates??

Delegate Types

Put simply, delegates are wrappers for function invocations. You can give it a function (or list of functions) with the same parameters and return type, and when invoked it will call all of them.

Let's see an example:

public class DelegateDemo { public delegate int SomeDelegateType(char param);   public int IntMethod(char param) { //whatever }   public void DelegateInvokeTest() { SomeDelegateType ourDelegate = IntMethod; ourDelegate.Invoke(); } }

Delegates allow for both generalization and anonymity of functions.

C# gives us 2 builtin types for delegates: Action<T1, T2, ...> and Func<TReturn, T1, T2, ...>.

  • Actions are wrappers for functions that accept 0 or more arguments and do not return a value. For example, void SomeMethod(int, char) could be represented as Action<int, char>.

    • Methods that don't have any arguments can be represented as a simple Action with no type arguments.

  • Funcs are wrappers for functions that accept 0 or more arguments and do return a value. For example, string SomeOtherMetohd(uint, short) could be represented as Func<string, uint, short>.

    • Like Actions, funcs that don't take arguments can be represented as Func<returntype>.

Both types can be created inline as lambdas:

// the arguments of an action are defined between the parentheses public Action<string> myAction = (str) => Console.WriteLine(str); public Func<int, long, long> myFunc = (long1, long2) => (int)(long1 + long2);

Anyway, let's get back to actually creating events.

public event Action ourEvent;

When you create an event, the class in which it is defined has all the power regarding when it's raised. For example, if you define ourEvent in class EventClass, it can call ourEvent?.Invoke();. However, a class NonEventClass cannot call EventClass.ourEvent?.Invoke();.

You can get around this by defining a public method that raises the event:

public static void ExternalRaiseEvent() => ourEvent?.Invoke();

Subscribing To Events

You need to give the events things to do when they're raised, which can be done with a single operator.

Action<string> someAction = (str) => { // whatever } ourEvent += someAction; // adds our action to the event's invocation list

Lambdas also work:

ourEvent += (str) => // inline whatever

You can unsubscribe from an event by using -= instead of +=:

ourEvent -= someAction // someAction will no longer be called

Subscribing and Unsubscribing From IL2CPP Events

While IL2CPP events work the same way, subscribing/unsubscribing from them is a little different.

Classes with events will have the events replaced with add_<eventname>(delegate) and remove_<eventname>(delegate) methods.

Remember how earlier we said that managed types aren't compatible with IL2CPP types? This applies to delegates as well. Luckily, Il2CppInterop gives us a function that allows us to convert.

Delegate Conversion (DelegateSupport class)

Say we have an event we want to subscribe to:

public void add_OnPlayerDied(PlayerEvent arg);

We can't pass our own function in raw because PlayerEvent is an IL2CPP delegate type. We need to convert it first:

var ourConvertedFunction = DelegateSupport.ConvertDelegate<PlayerEvent>(ourAction);

Now we can safely subscribe to the event.

add_OnPlayerDied(ourConvertedFunction); // no compiler errors :3

The same applies for the remove_ methods as well.
